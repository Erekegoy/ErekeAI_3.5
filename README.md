# 🤖 ErekeAI

Современный AI-ассистент для Android (Kotlin + Jetpack Compose) с поддержкой нескольких
AI-провайдеров (Gemini, Groq, OpenAI, Offline AI) и модульной архитектурой
(core / domain / data / features / ui / plugins).

Этот архив — полный **исходный код проекта**, готовый к открытию в Android Studio.
Собрать `.apk` можно только на машине с установленным Android SDK — в среде, где
готовился этот архив, доступа к сети и Android SDK не было, поэтому сама сборка
(`gradlew assembleDebug`) здесь не выполнялась. Ниже — как собрать за 5 минут.

## 📦 Что внутри

```
ErekeAI/
├── app/
│   ├── src/main/java/com/erekeai/
│   │   ├── app/            — Application, MainActivity, навигация
│   │   ├── core/            — DI (Hilt), безопасное хранилище ключей, утилиты

│   │   ├── data/             — Room (БД), DataStore (настройки), AI-провайдеры (Retrofit/OkHttp)
│   │   ├── features/chat/    — экран чата (Compose + ViewModel)
│   │   ├── features/settings/— экран настроек и API-ключей
│   │   ├── ui/theme/         — тема Material3
│   │   └── plugins/          — заготовка Plugin System
│   └── src/main/res/         — строки, цвета, темы, иконки
├── build.gradle.kts / settings.gradle.kts / gradle.properties
├── gradlew, gradlew.bat
└── local.properties.example
```

## 🚀 Как собрать

1. Установите **Android Studio** (Koala/2024.1+) с Android SDK 34, JDK 17.
2. Распакуйте архив и откройте папку `ErekeAI` через *File → Open* в Android Studio.
   Studio сама подтянет `gradle-wrapper.jar` и синхронизирует зависимости
   (нужен интернет при первом открытии — Studio скачает Gradle 8.7).
3. Скопируйте `local.properties.example` → `local.properties` и укажите путь к SDK
   (Studio часто делает это автоматически) и, при желании, свои API-ключи:
   ```
   sdk.dir=/Users/you/Library/Android/sdk
   GEMINI_API_KEY=ваш_ключ
   GROQ_API_KEY=ваш_ключ
   OPENAI_API_KEY=ваш_ключ
   ```
   Ключи не обязательны на этом шаге — их можно ввести прямо в приложении на
   экране «Настройки» (хранятся зашифрованно через `EncryptedSharedPreferences`).
4. Соберите и запустите:
   ```bash
   ./gradlew assembleDebug
   # или сразу на подключённое устройство/эмулятор
   ./gradlew installDebug
   ```

Если у вас нет Android Studio и вы хотите собрать чисто из терминала — нужен
установленный Android SDK и переменная `ANDROID_HOME`, после чего команды
`./gradlew` выше сработают напрямую (Gradle wrapper сам скачает нужную версию Gradle).

## ✅ Что уже реализовано в этом скелете

- Модульная архитектура `core / domain / data / features / ui / plugins`.
- Единый интерфейс `AiProvider` + `AiProviderRegistry` (AI Provider System) —
  реализации для **Gemini**, **Groq**, **OpenAI** (потоковая генерация через SSE)
  и заглушка **Offline AI**.
- Room-база (`ErekeDatabase`, `ChatDao`) — долговременная память диалогов.
- DataStore (`SettingsDataStore`) — хранение выбранного провайдера и темы.
- Безопасное хранилище API-ключей (`SecureKeyStore`, Android Keystore / AES256).
- Hilt DI (модули `AppModule`, `DatabaseModule`, `RepositoryModule`, `NetworkModule`, `ToolModule`, `AgentModule`).
- Экран чата на Jetpack Compose со стримингом ответов и историей.
- Экран настроек с выбором провайдера и вводом API-ключей.
- Заготовка Plugin System (`ErekePlugin`, `PluginRegistry`) для будущего расширения.

### 🧠 AI Agent + Tool Calling (новое)

Добавлен провайдеро-независимый **ReAct-агент** (`domain/agent`, `data/agent`):
модель рассуждает по циклу `Thought → Action → Observation → ... → Final Answer`
в свободном тексте (без привязки к нативному function calling конкретного API),
что позволяет агенту работать одинаково с Gemini, Groq и OpenAI.

- `AgentOrchestrator` / `AgentOrchestratorImpl` — сам цикл рассуждений (лимит шагов, защита от зависания).
- `ReActParser` — разбор ответа модели на `Thought` / `Action` / `Action Input` / `Final Answer`.
- `Tool` / `ToolRegistry` (domain) — контракт инструмента и его реестр; новый инструмент = один класс,
  ничего больше менять не нужно (регистрация — в `core/di/ToolModule.kt`).
- Готовые инструменты (`data/tools`):
  - **`web_search`** — поиск в интернете без API-ключа (HTML-версия DuckDuckGo + Jsoup).
  - **`list_files` / `read_file`** — чтение файлов из песочницы `agent_documents` внутри приложения
    (без опасных runtime-разрешений; путь защищён от path traversal).
  - **`calculator`** — безопасный вычислитель арифметики (собственный парсер, без eval).
  - **`current_datetime`** — текущая дата/время устройства.
- В `ChatScreen` — переключатель «Агент» в верхней панели; шаги агента (вызовы инструментов и их
  результаты) сохраняются в историю с новой ролью `Role.TOOL` и отображаются компактными карточками.
- `RunAgentTaskUseCase` — связывает агента с чатом: сохраняет каждый шаг в Room, чтобы история была
  видна и после перезапуска приложения.

⚠️ Формат ReAct через свободный текст — прагматичный компромисс для единого кода под три провайдера.
Следующий логичный шаг — перейти на нативный `tools`/`function_calling` API каждого провайдера
(структурированный JSON вместо парсинга текста), это надёжнее, но требует отдельной ветки на провайдера.

## 🧩 Vector Database + Документы (новое)

Первые два пункта из большого списка "чего не хватает" — сделаны по-настоящему, не заглушками:

### Векторная база данных (`domain/vector`, `data/vector`)
- `VectorStore` / `VectorStoreImpl` — локальная векторная БД поверх Room: чанки текста +
  их эмбеддинги (хранятся как `ByteArray`, 4 байта на float). Поиск — brute-force косинусное
  сходство по всем векторам одного провайдера эмбеддингов.
  ⚠️ Честно: это **не** специализированный ANN-индекс (Faiss/HNSW) — для личной коллекции
  документов на телефоне (сотни-тысячи чанков) более чем достаточно и быстро; для
  десятков тысяч+ чанков понадобится настоящий ANN-индекс, это отдельная более сложная задача.
- `EmbeddingProvider` — реальные эмбеддинги через **Gemini** (`gemini-embedding-001`) и
  **OpenAI** (`text-embedding-3-small`), нужен свой ключ (тот же, что для чата). Полностью
  оффлайн-эмбеддингов нет — это отдельная ML-модель, которую пришлось бы отдельно
  встраивать (как Gemma для LOCAL_LLM), сейчас не реализовано.
- Эмбеддинги одного документа всегда считаются одним и тем же провайдером; при поиске
  сравниваются только вектора из одного и того же провайдера (`embeddingProviderId`),
  чтобы не сравнивать вектора из разных пространств.

### Документы: PDF / DOCX / XLSX (`data/documents`)
- `PdfTextExtractor` — через PDFBox-Android (реальная библиотека, инициализируется в
  `ErekeAiApplication`). Работает с текстовыми PDF; отсканированные PDF без текстового
  слоя честно вернут пустой текст — OCR не реализован.
- `DocxTextExtractor` / `XlsxTextExtractor` — без тяжёлого Apache POI (плохо работает на
  Android — тянет `java.awt`/StAX): читают .docx/.xlsx напрямую как ZIP+XML через
  `XmlPullParser`. Реальный работающий парсер текста/ячеек, без таблиц-как-таблиц и картинок.
- `TextChunker` — режет длинный текст на чанки ~1200 символов с перекрытием, стараясь
  не рвать текст посреди предложения/абзаца.

### Новые инструменты агента
- `import_document` — извлекает текст из файла в песочнице `agent_documents`, режет на
  чанки, считает эмбеддинги, сохраняет в векторную БД.
- `knowledge_base_search` — семантический поиск по проиндексированным документам.
- `list_knowledge_base_sources` — какие документы уже проиндексированы.

### Новый экран «База знаний»
Реальный системный File Picker (SAF) для загрузки PDF/DOCX/XLSX/TXT/MD прямо из
интерфейса — раньше файлы в песочницу `agent_documents` вообще нечем было добавить.
Открывается через новую иконку 📖 на экране чата.

## 🤖 AI Developer Agent (новое)

Отдельная персона агента поверх того же `AgentOrchestrator`/`ToolRegistry` — специализация
под задачи разработки. Открывается новой иконкой 🤖 на экране чата (свой диалог, всегда
в режиме агента, с системным промптом персоны `AgentPersona.DEVELOPER`).

- **💻 Анализ проекта и структуры кода** — `analyze_project` (статистика по файлам/папкам/
  расширениям, TODO/FIXME, самые крупные файлы) и `code_search` (grep-подобный поиск по
  всем файлам песочницы с указанием файла и номера строки).
- **✍️ Генерация нового кода** и **🔧 Исправление ошибок и рефакторинг** — новый инструмент
  `write_file` (создание/перезапись/дозапись файла в песочнице, с защитой от path traversal);
  модель сначала читает файл через `read_file`, затем сохраняет новую версию.
- **🧪 Запуск проверок и анализ логов**:
  - `run_static_checks` — офлайн-эвристики по файлу (длинные строки, несбалансированные
    скобки, TODO/FIXME, пустые catch-блоки, висящие пробелы, смешение табов/пробелов).
    На телефоне нет доступа к gradle/lint/CI, поэтому это быстрый эвристический отчёт,
    а не полноценный статический анализатор.
  - `analyze_log` — детерминированный (без AI-вызова) разбор вставленного лога/стектрейса:
    тип и сообщение исключения, первая "своя" строка трейса, число `Caused by`, подсказка
    по десятку самых частых исключений (NPE, IndexOutOfBounds, ClassCastException, IO,
    OOM, NetworkOnMainThread и т.д.).
- **🐙 Работа с Git и GitHub** — `github_action` (owner/repo/action): `list_repo`, `get_file`,
  `list_issues`, `list_pulls`, `list_commits`, `create_issue` через GitHub REST API v3.
  На Android нет git-бинаря/терминала, поэтому вместо обёртки над командой `git` инструмент
  работает напрямую с REST API. Personal Access Token задаётся на экране «Настройки»
  (карточка «AI Developer Agent — Git/GitHub», хранится в `SecureKeyStore`); без токена
  доступны операции чтения публичных репозиториев с более жёстким лимитом запросов.
- **🧠 Планирование разработки** — `create_dev_plan` (цель, шаги, риски, критерии готовности)
  строит markdown-план и сохраняет его в `agent_documents/plans/`, чтобы агент мог
  ссылаться на него в следующих шагах ReAct-цикла, а пользователь — вернуться к нему позже.

Новое: `AgentPersona` (`domain/agent`) — переключает системную преамбулу промпта
(`GENERAL` для обычного чата, `DEVELOPER` для этого экрана); `RunAgentTaskUseCase` и
`AgentOrchestrator.run(...)` принимают персону как параметр, набор инструментов при этом общий.

Экран `features/devagent` (Compose + свой `DevAgentViewModel`) даёт быстрые действия-чипы
под каждый из сценариев выше — они либо сразу отправляют типовой запрос агенту, либо
подставляют шаблон в поле ввода, чтобы пользователь уточнил детали (например, какой файл
исправить или какой лог разобрать).

## 🔄 Что осталось по плану (делаем по одному, в порядке зависимостей)

- **RAG-оркестрация** — сейчас поиск по базе знаний доступен агенту как явный инструмент;
  следующий шаг — автоматическая инъекция релевантного контекста в каждый запрос (без
  необходимости агенту явно вызывать инструмент), реранжирование, цитирование источников.
- **Vision / Camera** — мультимодальные запросы к Gemini/OpenAI (анализ фото, доступ к камере).
- **Voice Agent** — голосовой ввод/вывод (STT/TTS), без wake word на первом этапе.
- **Local LLM через llama.cpp** — on-device инференс без сети (тяжёлая нативная интеграция,
  NDK-сборка).
- **MCP-клиент** — подключение внешних источников (Google Drive/Gmail/Calendar) как
  контекста для агента через протокол Model Context Protocol. GitHub уже подключён
  напрямую через REST API (см. раздел «AI Developer Agent» выше) — MCP не потребовался.
- **Multi-Agent** — несколько специализированных агентов (планировщик/исполнитель/критик)
  поверх существующего `AgentOrchestrator`.
- **Workflow Engine** — цепочки шагов с условиями, сохраняемые и повторно запускаемые.
- **Automation** — управление другими приложениями Android (Accessibility Service).
- **Plugin Marketplace** — вероятно, реалистично как каталог официальных инструментов
  с переключателями вкл/выкл, а не динамическая загрузка стороннего кода (это риск
  безопасности) — обсудим отдельно, когда дойдёт очередь.
- **API Server** — лёгкий встроенный HTTP-сервер на устройстве (например, NanoHTTPD/Ktor),
  чтобы дергать ErekeAI как бэкенд.
- **Cloud Sync** — нужен внешний бэкенд (например, Firebase) — потребует твоего аккаунта/проекта.

## ⚠️ Важно про API-ключи

Никогда не публикуйте `local.properties` или заполненные ключи в открытом репозитории —
файл уже добавлен в `.gitignore`. Для реальных ключей используйте либо
`local.properties`, либо экран «Настройки» в самом приложении (зашифрованное хранилище).
